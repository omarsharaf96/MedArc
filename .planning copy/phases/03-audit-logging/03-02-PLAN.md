  <name>Task 1: Inject audit writes into all 5 FHIR commands</name>
  <files>src-tauri/src/commands/fhir.rs</files>
  <behavior>
    For each of the 5 FHIR commands, the pattern after injection must be:
    1. middleware::check_permission — captures user_id (or fails before DB lock)
    2. db.conn.lock() — ONE lock acquisition
    3. Attempt FHIR operation — capture result (do not ? yet)
    4. write_audit_entry(&conn, ...) — within same lock scope
    5. Return result

    Specific behaviors:
    - create_resource: permission check returns _user_id — currently discarded; capture it.
      After INSERT, write audit with ActionType::Create, resource_type = input.resource_type,
      resource_id = created id, success = true.
      If INSERT fails, write audit with success = false, error_message = err.to_string().
    - get_resource: capture user_id from check_permission (currently _user_id — fix the underscore).
      Write audit with ActionType::Read, resource_type = resource.resource_type, resource_id = id.
    - list_resources: capture user_id. Write audit with ActionType::List,
      resource_type = resource_type.as_deref().unwrap_or("ALL"), resource_id = None.
    - update_resource: capture user_id from check_permission (currently discards).
      Write audit with ActionType::Update, resource_type = input.resource_type (add this field to
      UpdateFhirResource if not present; check models first), resource_id = input.id.
    - delete_resource: capture user_id. Write audit with ActionType::Delete,
      resource_type = None (not available in delete), resource_id = id.

    Critical: For ALL commands, the write_audit_entry call MUST be BEFORE the final `?` on
    the FHIR result. Pattern: let result = fhir_op(); write_audit_entry(..., result.is_ok(), ...)?;
    result (or result?)

    Add `device_id_state: State<'_, DeviceId>` parameter to all 5 commands.
    Use `&device_id_state.0` as the device_id argument to write_audit_entry.

    DeviceId is defined in src-tauri/src/audit/entry.rs (or a new src-tauri/src/audit/device.rs
    if preferred — keep it in entry.rs for simplicity).
  </behavior>
  <action>
    Modify src-tauri/src/commands/fhir.rs:

    Add import: use crate::audit::entry::{write_audit_entry, ActionType, DeviceId};

    For each command:
    1. Add `device_id_state: State<'_, DeviceId>` as the last parameter
    2. Change `let (_user_id, role)` to `let (user_id, role)` in check_permission
    3. Wrap FHIR operation in a let binding to capture result before returning
    4. After FHIR op (and after field filtering for get/list), call write_audit_entry within
       the conn lock scope
    5. Return the captured result

    For create_resource specifically: the FHIR op succeeds (conn.execute) or fails; capture
    both paths. After the execute succeeds, the resource is constructed — write audit with
    the created id. If execute fails, write audit with success=false before returning Err.

    Note for list_resources and get_resource: the conn lock is not currently held explicitly
    (they use conn.prepare/query_row directly). The conn MutexGuard IS held throughout because
    the lock() call assigns it to `conn`. Write audit within the same scope before conn drops.

    For the permission-denial case: when check_permission returns Err(Unauthorized),
    we do NOT have a DB connection yet. Write the audit entry in a separate DB lock
    acquisition AFTER the permission check fails:
    ```rust
    let perm_result = middleware::check_permission(&session, Resource::ClinicalRecords, Action::Read);
    if let Err(ref e) = perm_result {
        if let Ok(conn) = db.conn.lock() {
            let _ = write_audit_entry(&conn, "unknown", ActionType::Read,
                Some("FhirResource"), Some(&id), &device_id_state.0,
                false, Some(&e.to_string()));
        }
        return Err(perm_result.unwrap_err());
    }
    let (user_id, role) = perm_result.unwrap();
    ```
    Use "unknown" as user_id when session has no authenticated user.
  </action>
