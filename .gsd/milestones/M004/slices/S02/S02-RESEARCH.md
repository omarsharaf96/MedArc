# S02: Electronic Claims Submission (837P) — Research

**Date:** 2026-03-14

## Summary

S02 generates ANSI X12N 5010A1-compliant 837P EDI files from completed billing records and transmits them to Office Ally via SFTP. The 837P is the standard electronic claim format required by Medicare and virtually all commercial payers under HIPAA (45 CFR Part 162).

The primary technical challenge is 837P segment generation correctness. X12 5010A1 has strict loop/segment ordering and each loop has specific situational rules that differ between institutional (837I) and professional (837P) claim types. For PT, the 837P is always the correct form. The file must be generated by hand-building segments — the Rust `edi` crate handles parsing (835 ERA in S03) but has no write-path support.

Office Ally is the target clearinghouse. It accepts files via SFTP and returns 997/999 functional acknowledgements within minutes. The integration is SFTP-only (no REST API). Office Ally is the most widely used low-cost clearinghouse for solo PT practices.

The claim lifecycle (Draft → Validated → Submitted → Accepted → Paid/Denied) is tracked in `claim_index`. BillingStaff owns the lifecycle; Provider can generate and view claims.

**Confidence: MEDIUM** — X12 5010A1 format is well-documented but the PT-specific loop population (especially NTE segments, REF segments for prior auth, and the CLM05 facility/qualifier combinations) requires careful implementation. Office Ally SFTP connectivity requires real credentials to verify end-to-end.

## Recommendation

- Hand-build 837P segments as Rust string operations (no write-capable X12 library exists in Rust)
- Validate generated files locally using segment count and loop structure checks before upload
- Use the `ssh2` crate for SFTP (already planned in M004 CONTEXT)
- Store SFTP credentials in macOS Keychain via existing `keyring` crate
- Poll Office Ally `/edi/acks/` every 30 minutes for 997/999 acknowledgements
- Retire the SFTP risk in T02 by successfully transmitting a test claim to Office Ally sandbox

## ANSI X12N 5010A1 Format Structure (837P)

### Envelope Structure
```
ISA — Interchange Control Header (required, exactly 1)
  GS — Functional Group Header (required, exactly 1 per interchange)
    ST — Transaction Set Header (required, 1 per claim batch)
    BPR — Financial Information (optional, not used for claims)

    Loop 1000A — Submitter Name
      NM1*41 — Submitter (billing provider / practice)
      PER — Submitter contact info

    Loop 1000B — Receiver Name
      NM1*40 — Receiver (clearinghouse / payer)

    Loop 2000A — Billing Provider Hierarchical Level
      HL*1 — Billing Provider HL
      PRV — Provider taxonomy code

      Loop 2010AA — Billing Provider Name
        NM1*85 — Billing Provider (NPI, name)
        N3 — Billing Provider Address
        N4 — Billing Provider City/State/ZIP
        REF*EI — Tax ID (EIN or SSN)

      Loop 2010AB — Pay-to Address (if different from billing)

      Loop 2000B — Subscriber Hierarchical Level
        HL*22 — Subscriber HL (patient is subscriber for Medicare)
        SBR — Subscriber Information (payer, plan)

        Loop 2010BA — Subscriber Name
          NM1*IL — Insured Name (patient)
          N3 — Subscriber Address
          N4 — Subscriber City/State/ZIP
          DMG — Demographic Information (DOB, gender)

        Loop 2010BB — Payer Name
          NM1*PR — Payer (insurance company)
          N3 — Payer Address
          N4 — Payer City/State/ZIP
          REF*2U — Payer ID

        Loop 2300 — Claim Information (one per claim/encounter)
          CLM — Claim Information (claim ID, total charge, facility code)
          DTP*434 — Service Date (from/to)
          DTP*435 — Admission Date (not used for outpatient PT)
          CL1 — Institutional Claim Code (not used for 837P)
          REF*9F — Prior Authorization Number (from M003 auth_tracking)
          NTE — Claim Note (optional narrative)
          HI — Health Care Diagnosis Codes (ICD-10-CM)

          Loop 2310B — Rendering Provider
            NM1*82 — Rendering Provider (NPI)
            PRV — Provider taxonomy (PT taxonomy: 225100000X)

          Loop 2310C — Service Facility Location
            NM1*77 — Service Facility
            N3 — Facility Address
            N4 — Facility City/State/ZIP

          Loop 2400 — Service Line (one per CPT code)
            LX — Line Counter
            SV1 — Professional Service (CPT code, charge, units, modifiers)
            DTP*472 — Date of Service
            REF*6R — Line Item Control Number

    SE — Transaction Set Trailer
  GE — Functional Group Trailer
IEA — Interchange Control Trailer
```

### Key Segments for PT

**CLM05 (Place of Service / Frequency)**
- `CLM*<claim_id>*<total_charge>**<pos_code>:B:1`
- POS code for outpatient PT: `11` (office) or `22` (outpatient hospital)
- Third element: "B" = non-institutional, "1" = claim frequency (original claim)

**SV1 (Service Line)**
- `SV1*HC:<cpt_code>:<modifier1>:<modifier2>*<charge>*UN*<units>***<diagnoses>`
- Modifiers: GP (always for Medicare PT), then KX/CQ as applicable
- Up to 4 modifiers in SV1*02 field positions

**HI (Diagnosis Codes)**
- `HI*ABK:<icd10_code>*ABF:<secondary_code>`
- ABK = principal diagnosis (ICD-10-CM), ABF = secondary diagnoses
- Up to 12 diagnosis codes per claim

**REF*9F (Prior Authorization)**
- `REF*9F*<auth_number>` — required when auth number exists in M003 auth_tracking
- S02 reads `auth_index` to populate this segment

## Clearinghouse Options

### Office Ally (Primary)
- Most widely used for solo PT; free for providers
- SFTP upload to `ftp.officeally.com` (port 22)
- Upload path: `/edi/claims/`; acknowledgement path: `/edi/acks/`; ERA path: `/edi/era/`
- File naming: `<date_yyyymmdd>_<time_hhmmss>_<submitter_id>.837`
- Payer enrollment required for each payer before claims can be routed
- Returns 999 (for 5010) functional acknowledgements within 2–4 hours
- Real-time claim status via their web portal; no API for status queries (SFTP only)

### Availity (Backup / Reference)
- Larger enterprise clearinghouse; REST API available
- More payer connections; supports eligibility verification (ELIG-01 in future milestone)
- Higher cost for solo practices
- Not the primary target for M004 but the `payer_config` schema supports multiple clearinghouses

## Rust `edi` Crate Capabilities

The `edi` 0.4 crate provides:
- `edi::parse(input: &str) -> Result<EdiDocument, EdiParseError>` — tokenises an X12 document into segments and loops
- `EdiSegment::get_element(index)` — element access by position
- Handles ISA/GS/ST envelopes; recognises loop delimiters

**Limitations for 837P generation:**
- No write path — cannot construct segments programmatically
- No schema validation — does not enforce X12 5010A1 loop structure
- Best used for 835 ERA parsing in S03

**837P Generation Approach:**
Build segments as Rust string formatting. Define constants for element separators (`*`) and segment terminators (`~`). Create a `ClaimBuilder` struct that accumulates segments into a `Vec<String>` and serialises with `segments.join("~\n")`.

```rust
pub struct ClaimBuilder {
    segments: Vec<String>,
    interchange_control_number: u32,
    group_control_number: u32,
    segment_count: u32,
}

impl ClaimBuilder {
    pub fn isa_header(&mut self, submitter_id: &str, receiver_id: &str) { ... }
    pub fn loop_1000a_submitter(&mut self, billing: &BillingProvider) { ... }
    pub fn loop_2000b_subscriber(&mut self, patient: &PatientRecord) { ... }
    pub fn loop_2300_claim(&mut self, claim: &ClaimInput) { ... }
    pub fn loop_2400_service_line(&mut self, entry: &CptEntry, line_num: u32) { ... }
    pub fn build(self) -> String { ... }
}
```

## SFTP Upload Workflow

```
1. Generate 837P file → stored as temp file in app support dir
2. Validate file (local segment count check + required segment presence)
3. Open SFTP connection to ftp.officeally.com:22
   - Authentication: private key from macOS Keychain (stored during setup)
   - known_hosts: Office Ally fingerprint verified on first connection
4. Upload file to /edi/claims/<filename>.837
5. Update claim_index status: Submitted, store submitted_at timestamp
6. Background task polls /edi/acks/ every 30 min for 997/999 files
7. On 997/999 receipt: parse acknowledgement, update claim_index to Accepted or Rejected
```

### SFTP with `ssh2` Crate
```rust
use ssh2::Session;
use std::net::TcpStream;
use std::path::Path;

pub fn upload_claim_sftp(
    host: &str,
    username: &str,
    private_key_path: &Path,
    local_file: &Path,
    remote_path: &str,
) -> Result<(), AppError> {
    let tcp = TcpStream::connect(format!("{}:22", host))?;
    let mut sess = Session::new()?;
    sess.set_tcp_stream(tcp);
    sess.handshake()?;
    sess.userauth_pubkey_file(username, None, private_key_path, None)?;
    let sftp = sess.sftp()?;
    // upload file bytes
    Ok(())
}
```

## Claim Lifecycle States

```
Draft       — claim generated from billing record, not yet validated
Validated   — local validation passed (all required segments present, no obvious errors)
Submitted   — uploaded to clearinghouse SFTP, submitted_at timestamp set
Accepted    — 999 AA (Accepted) received from clearinghouse
Rejected    — 999 AK (Rejected) received; rejection reason stored in claim_index
Paid        — ERA posting (S03) sets this when payment received
Denied      — ERA posting (S03) sets this when denial received
Appealed    — manual override by BillingStaff after denial review
```

State transitions are enforced in `update_claim_status`. Invalid transitions return `AppError::Validation`.

## Common Pitfalls

- **ISA control numbers must be unique per interchange** — Use a monotonically incrementing counter stored in the `payer_config` table. Reusing a control number causes the clearinghouse to reject the interchange as a duplicate.
- **Segment count in SE must be exact** — `SE*<count>*<control_number>` where count includes ST and SE themselves. Off-by-one on segment count = invalid transaction set.
- **NM1 qualifier for Medicare** — Medicare's EDI receiver ID (NM1*40) qualifier for Office Ally routing is `46` (Electronic Transmitter Identification Number) with ID = `OFFICEALLY`. The payer NM1 uses `PR` qualifier with payer_id = `MEDICAR` (Office Ally's Medicare routing ID for Part B).
- **CLM05 facility type must match POS code** — For outpatient PT in an office setting: `11:B:1`. Getting the colon-delimited composite elements wrong causes parse errors at the clearinghouse.
- **Taxonomy code for PT** — Physical Therapist taxonomy is `225100000X`. Required in PRV segment. Without it, Medicare will reject the claim.
- **GP modifier positioning** — In SV1, modifiers are in positions SV1-02 through SV1-05. GP must appear first (highest priority position). KX and CQ follow. Maximum 4 modifiers per SV1.
- **ICD-10 coding qualifier** — HI uses qualifier `ABK` for primary diagnosis and `ABF` for secondary. Using wrong qualifiers (e.g., `BK` from the older 4010 format) causes 999 rejection.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| 837P segment generation correctness | High | Validate generated file with WEDI free online validator before S02 is marked complete; unit tests assert segment presence and ordering |
| Office Ally SFTP connectivity | High | Retire in T02 with a real test transmission to Office Ally sandbox environment using test credentials |
| ISA envelope control number management | Medium | Store last-used control number in `payer_config`; increment atomically within a DB transaction |
| Clearinghouse payer enrollment lag | Medium | Payer enrollment for Medicare can take 2–4 weeks; document that enrollment is a prerequisite for live billing, not a MedArc defect |
| `ssh2` crate known_hosts management | Medium | On first connection, present fingerprint to user for verification; store accepted fingerprints in SQLCipher |
| Claim rejection from invalid ICD-10 codes | Low | Validate ICD-10 code format (regex: `^[A-Z][0-9]{2}(\.[A-Z0-9]{1,4})?$`) before claim generation |
